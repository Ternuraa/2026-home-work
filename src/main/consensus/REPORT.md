## 3. Анализ времени 
| PING_TIMEOUT | Среднее время до нового лидера | Ложные выборы |
|--------------|--------------------------------|---------------|
| 1000         | нет данных                     | -             |
| 3000         | 2088                           | нет           |
| 5000         | 4069                           | нет           |

### Вывод
При PING_TIMEOUT = `1000 мс` консенсус не был достигнут ни в одном из пяти прогонов. Причина в том, что в синхронной модели с мгновенной передачей сообщений таймаут оказался меньше, чем время, необходимое для полного цикла выборов. Узлы начинали новые выборы до завершения предыдущих, что приводило к бесконечной гонке перевыборов.

При PING_TIMEOUT = `3000 мс` и `5000 мс` консенсус достигался хотя бы в одном из прогонов. Среднее время указано только по успешным замерам. При 3000 мс оно составило `2088 мс`, при `5000 мс – 4069 мс`. Видна прямая зависимость: увеличение таймаута пинга ведёт к пропорциональному росту времени обнаружения отказа лидера и, как следствие, к более долгому восстановлению.

Ложные выборы в данном эксперименте не фиксировались, поскольку случайные отказы были отключены, а падение лидера производилось принудительно. В реальной сети с ненулевыми задержками при малых таймаутах возможны ложные срабатывания.

Вывод: для стабильной работы алгоритма в условиях мгновенной передачи сообщений минимально приемлемый PING_TIMEOUT составляет около `3000 мс`. В production‑подобной среде его следует увеличить с учетом сетевой задержки до 5000 мс и более, а также доработать механизм предотвращения повторных выборов.
